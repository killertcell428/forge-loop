## こんな経験、ないですか?

個人で OSS を作っていて、こういう状態になったこと、ないですか:

- 久しぶりに自分の repo を開いたら、issue が 30 件溜まっている
- 「修正したい」と思いつつ仕事や生活で手が回らない
- 半年前の release から先に進んでいない
- メンテナンスが楽しくない、義務感だけになっている

私もそうでした。**Python 製の小さな OSS を作って公開したけど、release が止まりがち**。

## やってみたこと

そこで、**Claude Code を 6 時間おきに自動で起動して、自分の OSS を勝手に改修してもらう仕組み**を組んでみました。

結果はこうなりました:

- **5 日間で 17 回の改修サイクルが回った**
- **PyPI に v1.0.1 → v1.0.12 まで 11 回リリースされた**
- 朝起きたら、自分が一切タッチしていないバージョンが世の中に出ている
- ルールが 137 個増え、テストも一緒に増えた

「え、勝手にリリースされちゃうの? 怖くない?」と思いますよね。私も最初はそう思いました。
実は、**そのまま AI に丸投げすると確かに事故ります**。これも後で具体例で書きます。

仕組み自体はシンプルで、図にするとこうです:

![6時間おきに動く仕組み](https://raw.githubusercontent.com/killertcell428/aigis/master/images/forge-loop/loop.png)

6 時間ごとに「調査 → 計画 → 実装 → テスト → リリース」を順番にやるだけ。**人がやっている作業を、そのまま AI に置き換えただけ**です。

## 朝起きたら何が起きていたか

5 月 11 日の朝。私が寝ている間にこんなことが起きていました (実際のログを抜粋):

```
朝 3:00  Claude Code が自動起動
朝 3:00  「今日は multi-agent 関連の調査をする番だな」と判断
朝 3:00  arxiv / セキュリティ企業のブログから新しい攻撃を 7 件発見
朝 3:15  「この 3 件は実装できそう、1 件は大きすぎるから保留」と仕分け
朝 3:20  実装ファイルに regex を追加 (40 行)
朝 3:25  テストを 8 個追加 → 1222 件全部通る
朝 3:30  累積 3 件超えたので v1.0.12 としてリリース判定
朝 3:32  GitHub Actions が tag を検知 → PyPI に自動公開
朝 3:34  終了。ログを 1 行追記して、寝る。
```

私は朝、PyPI のリリース通知メールで気付きました。

**これを 5 日間で 17 回繰り返した**結果が、冒頭の「11 回リリース」です。

## 「AI に任せたら暴走するんじゃ?」

その通りです。**最初の数回は実際に暴走しました**。

私が経験した失敗:

| 何が起きたか | どうなったか |
|---|---|
| 同じ題材ばかり 3 回連続で触った | 流行のテーマに引きずられる、退屈な改修ばかり |
| 「考えました」だけで何も実装しないサイクル | 何が起きたか後から追えない |
| 大きな API 変更を勝手に commit して merge | リポジトリが一時的に壊れた |
| 既に直したバグを 2 回直そうとした | 履歴を読まずに作業 |

これらを **「次は AI が暴走する前提」で防ぐ設計**にしたら、安定して回り始めました。

## 暴走させない 3 つの工夫

肝は **「外枠は人が決めて、中身だけ AI に任せる」** ことです。

![境界線設計](https://raw.githubusercontent.com/killertcell428/aigis/master/images/forge-loop/boundary.png)

### ① 今日やることリスト を人が事前に作っておく

家のおかず当番表みたいに、**「今日はこの分野だけ手を入れる」を人が決めておく**。AI に「今日何やる?」と聞かない。

具体的にはファイル 1 個で済みます:

```yaml
# 改修したい分野の一覧 (例: ドキュメント生成 OSS の場合)
domains:
  - id: 0
    name: api-reference        # API リファレンス
  - id: 1
    name: getting-started      # 入門ガイド
  - id: 2
    name: troubleshooting      # トラブルシューティング
  - id: 3
    name: tutorials            # チュートリアル
  - id: 4
    name: faq                  # よくある質問
```

毎回、**順番に 1 個ずつ**消化していく。これだけで「流行に流される」「楽な題材ばかりやる」を防げます。

### ② 各段階のメモを必ずファイルに残させる

AI に「考えただけ」「やったけど記録なし」を許さない。

```
改修フォルダ/
├── 調査メモ/  ← AI が何を調べたか
│   └── 2026-05-11_03-00.md
├── 変更内容/  ← AI が何を実装したか
│   └── 2026-05-11_03-00.md
└── 保留リスト/ ← AI が「やらない」と判断したもの
    └── 2026-05-08_大きすぎる変更.md
```

**「ファイルが残ってないと次の段階に進めない」というルールを仕組みで強制**します。これで AI の作業が後から完全に追えるようになります。

「物質化」という小難しい言葉を使いがちですが、要は **「言ったらメモ取ってね」を機械にやらせる** だけです。

### ③ 怪しい変更は「保留フォルダ」に退避させる

AI に「やらない判断」を許す。これが地味に効きます。

```
こういう変更は AI が自分で判断して "保留フォルダ" に入れる:
  - 100 行を超える変更
  - 公開 API を変えてしまう変更
  - 既存のテストを書き換える必要がある変更
  - AI 自身が「自信ない」と感じた変更
```

実際、私の OSS では **22 件**がこの保留フォルダで眠っています。
週末にまとめて読んで、「これは実装、これは却下、これは放置」と人間が判断します。

「保留できる」と分かっていると、AI は **「とにかく実装する」に倒れずに**、危なそうなものをちゃんと避けてくれます。

## 試してみたい人へ (3 ステップ)

仕組みはテンプレートとして [GitHub に公開](https://github.com/killertcell428/forge-loop) しています。3 ステップで動かせます。

### ステップ 1: clone する

```bash
git clone https://github.com/killertcell428/forge-loop
```

### ステップ 2: 自分の OSS の "改修したい分野" を 5〜10 個書く

`template/auto-improvement/domains.yaml` を自分の OSS 用に書き換えるだけです。

```yaml
domains:
  - id: 0
    name: あなたの OSS の分野 1
  - id: 1
    name: 分野 2
  # ... 5〜10 個
```

### ステップ 3: GitHub Actions のスケジュール設定をコピー

`template/.github/workflows/forge-cycle.yml` を自分の repo にコピーして、`ANTHROPIC_API_KEY` を Secrets に入れる。これだけです。

これで **6 時間ごと**に自分の OSS が勝手に改修されはじめます。

詳しい設計や、もっと深い話は [リポジトリの docs/](https://github.com/killertcell428/forge-loop/tree/main/docs) を見てください。

## どんな OSS に向くか

「N 個の改修分野があって、それぞれを継続的に育てたい」という構造の OSS なら何でも合います。例:

| OSS の種類 | "分野" の例 |
|---|---|
| ドキュメント生成系 | チュートリアル / API リファレンス / FAQ / 入門 / 移行ガイド |
| Linter / Formatter | 言語別 (Python / TS / Go) / 規約別 / セキュリティ系 |
| CLI ツール | サブコマンド別 / エラーメッセージ / ヘルプ文 |
| テスト生成 | モジュール別 / エッジケース別 |
| 多言語 i18n | 言語別 |

逆に **「1 つの大きな機能を一気に書きたい」** タイプは向きません。それは人がやる仕事です。

## 何が新鮮なポイントか

正直なところ、技術スタック自体は普通です: **Claude Code + GitHub Actions + 1 個の yaml**。難しいプラットフォームも、特別な AI もいりません。

ただ、**「AI に裁量をどこまで与えるか」の設計**がほぼ全てで、ここを間違えると暴走するし、ここをきちんと設計すると **長期間 任せられる**ようになります。

ポイントを整理すると:

> AI に「自由」を与えると暴走する。
> AI に「厳格な脚本」を渡すと無価値になる。
> **AI には「外枠」を渡して、その中で裁量を与える** のがちょうどいい。

3 つの外枠:
- **何をやるか** は人が決める (おかず当番表)
- **どう残すか** は人が決める (メモのフォーマット)
- **どこで止めるか** は人が決める (保留に退避するルール)

これだけで AI は安全に走り続けます。

## 補足: 実際に動かしている OSS の話

ここまで例として書いていた OSS は、私が個人で作っている **[Aigis](https://github.com/killertcell428/aigis)** という Python 製の LLM セキュリティスキャナです。

実際の運用ログ・調査メモ・保留フォルダの中身は **そのまま公開** していて、[ここで読めます](https://github.com/killertcell428/forge-loop/tree/main/examples/aigis/snapshot/)。

ちなみに、この **「AI に OSS の継続改善を任せる仕組み」自体には名前を付けました**: **Forge Loop**。
「鍛冶 (forge) のように毎日少しずつ叩き続ける」イメージです。気に入ってもらえたら使ってみてください。

## まとめ

- **個人 OSS のメンテに時間が割けない人へ**: Claude Code を 6 時間おきに動かすと、自分の OSS が勝手に育っていきます
- **「AI に丸投げで暴走しないか」が心配な人へ**: 「外枠 (今日やること + メモ規約 + 保留ルール) は人が決める」だけで安全に回せます
- **試しやすさ**: GitHub からテンプレートを clone して、YAML を 1 個書き換えて、GitHub Actions に 1 ファイル置くだけです
- **私の OSS の実例**: 5 日間で 11 回リリース、22 件が保留フォルダに眠り、人レビュー待ちで止まっています

「自分でメンテする時間がないけど、消えてほしくない OSS」がある人は、一度試してみてください。
