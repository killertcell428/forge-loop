# AI に OSS の継続改修を任せる時、「自由を与える」のではなく「外枠を渡す」設計パターン — Forge Loop

> **Zenn 本編の派生記事。** より「エンジニア向けの抽象論」に寄せ、コードや事例は薄く、設計思想を厚く扱う。
> Qiita / note / 自前ブログ向け。

---

## 動機

ここ 1〜2 年、「AI に自動 PR を作らせる」「AI でリリースノートを書かせる」系の事例は増えてきた。
しかし、**「AI に OSS の継続的な機能追加を全任せして、半自動でリリースし続ける」事例**はまだ少ない。

私は LLM セキュリティスキャナ Aigis でこれを 27 サイクル運用していて、設計から見えてきたことがある。
**これは技術問題ではなく、境界線設計の問題**だった、ということ。

この記事では、技術スタック (Claude Code / GitHub Actions) の話ではなく、**「AI と人の責任をどう切り分けるか」**の話を書く。

---

## 失敗からスタート

最初に試したのは「Claude Code に CLAUDE.md で『継続的に改善してね』と書いて 6h ごとに起動する」。

結果:
- 3 サイクル連続で同じドメインを処理した
- 「考えました」だけで何も残らない cycle が混じった
- 200 LOC の PR が勝手に merge されて事故った
- 同じバグを 2 回直していた

「AI に自由を与える」は **物理的にうまくいかない**。
かといって、「ファイル X の Y 行を Z に変えろ」みたいな脚本を書くのは、AI を使う意味がない。

中間が必要だった。

---

## Forge Loop の主張

> **「外枠は人が決め、中身は AI に任せる。」**

これだけ。

| 層 | 誰が決める | 内容 |
|---|---|---|
| 外枠 | 人 | ドメイン一覧 / 段階の規約 / 保留ルール / リリース判定 |
| 中身 | AI | 各ドメインで何を調べる / どう実装する / どのテストを足す |

「外枠」と「中身」の切り分けに3つの軸がある。

---

## 3 つの軸 (抽象論)

### 軸 1: WHAT — 「何をやるか」は AI に選ばせない

メタ判断 (「今日は何をやるか」) は AI が最も暴走しやすい場所。
理由は単純で、メタ判断には **過去サイクルとの一貫性**が要求されるが、AI は文脈窓に依存するので、長期一貫性が原理的に弱い。

→ ドメイン一覧を人が手で書き、巡回ロジック (mod N) で**機械的に**選ぶ。
   AI には「ドメイン**内**で何をやるか」だけを判断させる。

### 軸 2: HOW — 思考を必ずファイルに落とす

「考えただけ」「やったけど記録なし」は監査不能。
プロンプトで「Markdown で記録すること」と書くだけだと守られない。

→ 構造で強制する。例: **`research/<cycle_id>.md` が存在しないと plan に進めない**、というように、ファイルの存在を次段階の前提条件にする。

### 軸 3: STOP — 「やらない判断」を機械的に

AI に「これは大きすぎるから pending にしておく?」と聞くと、ほぼ「いけそうな気がします」と答える。
やめる判断を AI に委ねるのは **構造的に難しい**。

→ LOC カウントなどの**機械的指標**で判定する。
   AI には判定の **実行**だけさせて、判定の **基準**は人が握る。

---

## なぜこれが効くのか

3 軸すべてに共通する原理は同じ:

> **AI に任せていいのは「中身を作る判断」だけ。「形式を決める判断」「メタ判断」「やめる判断」は人が握る。**

AI は中身の生成は得意だが、以下が苦手:
- 長期一貫性 (文脈窓を超えた整合性)
- 自分の能力の限界判定 (「これは無理」と認める)
- 多目的最適化 (品質 × 速度 × リスク のバランス)

これらは **人が設計図を描いて、AI に従わせる**のが正解。

逆に、AI は以下が得意:
- 特定のドメインを与えられた時の調査・実装
- パターン認識 (regex / similarity)
- 既知の設計図への準拠

「得意なところ」だけ任せて、「苦手なところ」は外枠で塞ぐ。これが Forge Loop。

## 一般化: AI エージェント設計の指針として

Forge Loop は「OSS の継続改修」を例にしているが、もっと一般的な指針が抽出できる:

### 指針 A: メタ判断を AI から奪う

「何をやるか」を AI に決めさせる場面では、**決定論的なロジックを噛ませる**。
例:
- ドメイン巡回 (mod N)
- 優先度キューからの取り出し
- 外部システムからの指示受け取り

### 指針 B: 思考を artifact に着地させる

エージェントの推論結果を **必ずファイル / レコード**に残す。
例:
- 推論ログを <cycle_id>.md に保存
- 決定を audit table に insert
- 提案を ticket として外部システムに登録

### 指針 C: 「やめる判断」を機械化する

エージェントに自由意志でやめさせない。
例:
- リソース使用量しきい値
- LOC / 行数しきい値
- 確信度スコア

### 指針 D: 失敗を append-only で残す

成功も失敗も同じ列で記録。**失敗を削除/隠蔽できない設計**。

### 指針 E: 配布段階を別パイプラインに分離

エージェントは「artifact 生成」までを担当。
**配布パイプライン (CI/CD)** は AI 関与外、独立に動く。
これで「AI が壊しても配布されない」が成立する。

## 適用領域

OSS の継続改修以外にも、以下のような領域に同じ設計が使えるはず:

| 領域 | ドメイン例 |
|---|---|
| 社内ドキュメント整備 | 章 / 部署 |
| データセット品質管理 | カテゴリ / ラベル |
| マーケティングコンテンツ生成 | プロダクト機能 / 顧客セグメント |
| Q&A bot のナレッジ拡張 | カテゴリ / 製品 |
| ブログサイトの自動運営 | テーマ |
| バグ修正の自動化 | コンポーネント |

共通するのは **「N 個のドメインを継続的に育てたい」**シチュエーション。

## 実装の話 (おまけ)

Claude Code + GitHub Actions で動かしている。技術スタックは:

- Claude Code Plugin として skill / agent を配布
- GitHub Actions cron で 6h ごとに起動
- 状態管理は git でファイルとして
- リリースは既存 release.yml に乗せる

詳細は [Zenn 本編記事](https://zenn.dev/...) と [forge-loop リポジトリ](https://github.com/...) を参照。

---

## まとめ

- **AI に「自由」を与えると暴走する。**
- **AI に「脚本」を渡すと無価値になる。**
- **AI には「外枠」を渡して、中身の裁量を与える。**
- 外枠は「メタ判断」「形式」「やめる判断」の 3 つを人が握る。
- 中身の生成だけ AI に任せる。

この設計が成立すると、**AI に長期間任せられる**ようになる。
