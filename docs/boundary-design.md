# Boundary Design — AI と人の境界線をどう引くか

Forge Loop の **真の主張**はここ。
「巡回 / 物質化 / 保留」は手段で、目的は **AI と人の境界線設計**。

---

## 大前提

AI に **自由を与える**と暴走する。
AI に **厳格な脚本**を与えると無価値になる。

Forge Loop は中間を取る:

> **「外枠」は人が決め、「中身」は AI に任せる。**

問題は「外枠」と「中身」をどう切り分けるか。

---

## 切り分けの 3 軸

### 軸 1: **WHAT (今日何をやるか)** — 必ず人

| 誰が決める | 内容 |
|---|---|
| **人** | ドメイン一覧 (domains.yaml) を書く |
| **人** | ドメインの順番 / 巡回ロジック (mod N) を決める |
| **AI** | 該当ドメイン**内**で具体に何をやるか |

**理由**: メタ判断 (「今日は何をやるか」) は、AI が最も暴走しやすい場所。流行・直前文脈・楽な題材に簡単に流れる。決定論的に固定すべき。

### 軸 2: **HOW (思考の残し方)** — 必ず人

| 誰が決める | 内容 |
|---|---|
| **人** | 各段階の出力ファイル名規約 |
| **人** | ディレクトリ構造 (research/ → changes/ → pending/) |
| **AI** | 各ファイルの中身 |

**理由**: 「考えただけ」「実装したけど記録なし」は監査不能。形式を人が決めて強制することで、AI を **必ずファイルに着地させる**。

### 軸 3: **STOP (やらない判断)** — 必ず人

| 誰が決める | 内容 |
|---|---|
| **人** | pending 行きの閾値 (LOC, API breaking, ...) |
| **人** | pending を実装に戻すかの最終判断 |
| **AI** | 閾値に照らして pending 行きかを **機械的に**判定 |

**理由**: AI に「やめる判断」を任せると、「いけそうな気がします」と実装に倒れる。閾値は人が決めて、AI には機械的なチェックだけさせる。

---

## やってしまいがちなアンチパターン

### アンチ 1: WHAT を AI に決めさせる

> 「OSS を良くしておいて。何をやるかも任せる。」

→ 流行追従 / 一点特化 / 全部後回し になる。
→ 解: ドメイン一覧を人が書いて、巡回ロジックで強制。

### アンチ 2: 物質化を「努力義務」にする

> プロンプトに「思考過程を Markdown で残すこと」と書く

→ 守ったり守らなかったり。長期間で蒸発する。
→ 解: 「research/<cycle_id>.md が存在しないと plan に進めない」と **構造で**強制する。

### アンチ 3: pending 行きの基準を AI に判断させる

> 「破壊的変更っぽかったら pending に入れて」

→ 「破壊的変更っぽくない」と AI が判断して実装に進む。
→ 解: LOC カウントなど **機械的指標**で判定する。

### アンチ 4: リリース判定を AI に任せる

> 「いい感じになったら release してね」

→ 1 件で release してしまう / 50 件溜め込んで release しない / 判断が回ごとに揺れる。
→ 解: `min_accumulated_units` のような明示しきい値で。

### アンチ 5: 失敗を隠す

> プロンプトに「失敗は最小化すること」と書く

→ AI が失敗を「成功」と記録する。
→ 解: 各サイクルの結果を INDEX.md に **追記式**で残す。書き換え禁止。

---

## 境界線の引き方の判断基準

ある判断を **AI に任せていいか**は、こう問う:

1. **その判断は、過去サイクルと一貫性が必要か?** Yes → 人が決める
2. **その判断は、流行や文脈で簡単に揺れるか?** Yes → 人が決める
3. **その判断の誤りはリポジトリを破壊しうるか?** Yes → 人が決める
4. **その判断は、機械的指標で代替できるか?** Yes → 機械的指標で
5. **その判断は、上記いずれにも当たらないか?** Yes → **AI に任せていい**

---

## 検証: Aigis の実例

| 判断 | 誰が | なぜ |
|---|---|---|
| 「今日は supply-chain-attacks ドメイン」 | 人 (ROTATION.md) | メタ判断、決定論で固定 |
| 「LangChain CVE-2025-68664 を実装」 | AI | ドメイン内の中身、AI 裁量 |
| 「regex はこの形」 | AI | 実装詳細、AI 裁量 |
| 「200 LOC 超だから pending」 | 機械的指標 | LOC カウント |
| 「累積 3 件で release」 | 機械的指標 | エントリ数カウント |
| 「リリースノートに rule_id を必ず書く」 | 人 (skill 定義) | 監査可能性のため |

---

## まとめ

| AI に任せる | 人が引く線 |
|---|---|
| ドメイン**内**の何を調べるか | ドメイン**一覧** |
| 各ファイルの**中身** | ファイル名**規約** |
| 閾値**判定の実行** | 閾値**値そのもの** |
| 中身を作る | 形式を決める |

**「中身は自由、形式は厳格」。** これが Forge Loop の境界線設計。
