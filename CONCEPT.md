# Forge Loop — Concept (North Star)

> このドキュメントは Forge Loop プロジェクトの「北極星」。
> 記事・README・Plugin 設計などすべての成果物は、この文書を起点に書かれる。

---

## 1行で

**Claude Code に、OSS の継続改善を「数時間おきに 調査 → 実装 → GitHub リリース」まで全任せできるループを作った。鍵は AI と人の境界線をどう引くか。** Aigis で 27 サイクル / 137 ルール / v1.0.7 → v1.0.17 まで運用中。

---

## なぜ作ったか

AI に「OSS を良くしておいて」と丸投げすると、こうなる:

- 毎回同じドメインばかり触る（直近の流行 / 楽な題材に偏る）
- 「考えました」だけで何も残らない（思考の物質化がない、監査できない）
- 調子に乗って巨大 PR や API 破壊変更を勝手にやる
- どこで止めるか分からず無限ループする

逆に、AI に厳格な脚本を渡すと **何もできなくなる**（決まりきった改修しか出力しない）。

Forge Loop が解いた問題は: **「AI に裁量を与えつつ、暴走させない外枠」をどう設計するか**。

---

## 主役の主張

**外枠を人が決め、中身を AI に任せる。**

| 層 | 誰が決めるか | 内容 |
|---|---|---|
| **外枠** | 人 | ドメイン一覧 / 段階の規約 / 保留ルール / リリース判定 |
| **中身** | AI | 各ドメインで何を調べる / どう実装する / どのテストを足す |

これが「人と AI の境界線」。AI に「中身の裁量」だけを与えて、「枠の設計権」は人が握る。

---

## 大事な点 3 つ（境界線設計の具体テクニック）

### ① 巡回 (Rotate) — 「何を改善するか」を AI に選ばせない

N 個のドメインを mod N で順番に消化する。

- 1 ドメイン特化を防ぐ
- 「今回は何やる？」のメタ判断を AI から奪うことで、判断ループの暴走を防ぐ
- ドメイン一覧は人が手で書く（=人が境界を引く場所）

**Aigis 実体**: `auto-improvement/ROTATION.md` の `NEXT_INDEX` (mod 10) + 10 セキュリティドメイン

### ② 物質化された段階 (Materialize) — AI の思考を必ずファイルに落とす

各段階の出力が必ずディレクトリに残る規約。

- `research/YYYY-MM-DDTHH-MM_NN-<domain>.md` — 調査結果
- `changes/YYYY-MM-DDTHH-MM.md` — 実装案
- 実装コード (通常通り)
- テスト (通常通り)
- `CHANGELOG.md` 追記

**「考えただけ」「実装したけど記録なし」を構造的に不可能にする。** 後から人が読めばどの段階で何が起きたか分かる。これがあるから AI に長期間任せられる。

### ③ 保留レーン (Defer) — 完了 / 失敗の二択にしない

破壊的 / 大規模 / 不確実な提案は実装せず、`pending/` に隔離する。

- ≥100 LOC → pending
- API breaking → pending
- 確信度が低い → pending

**「やらない判断」を AI に許す**。Aigis の `pending/` には実際に 9 件積まれていて、人が触るまで眠っている。これがないと AI は「とにかく実装する」に倒れて事故る。

## 1サイクルの実況 (Aigis 実例 2026-05-13 03:00 UTC)

```
03:00:00  cron が Claude Code を起動
03:00:05  ROTATION.md を読む → NEXT_INDEX: 6 → "supply-chain-attacks"
03:00:30  research/...md 作成 → arxiv/CVE を漁って 3 件発見
03:15:00  changes/...md に実装案を記録
03:20:00  patterns.py に regex を 3 件追加
03:25:00  tests/ にテスト 4 件追加 → pytest 通る
03:35:00  4 件目アイデアは 200 LOC 超 → pending/ へ隔離
03:40:00  累積 ≥3 件で CHANGELOG 追記 + git tag v1.0.13
03:45:00  GitHub Actions が tag → PyPI 自動公開
03:45:30  INDEX.md に 1 行追記 → エージェント終了
```

これを **6 時間ごとに 27 回**繰り返した結果が v1.0.7 → v1.0.17, 137 ルール。

## 何が新規性か

執筆時点 (2026-05) で、AI に OSS 改修を本当に「任せている」事例は稀:

| よくある事例 | Forge Loop |
|---|---|
| AI が PR をレビューする | AI が PR を作って merge して release する |
| AI がドキュメント生成 | AI が機能を増やしてリリースする |
| 単発タスク (1回限り) | 数時間おきに 27 回続く |
| 多エージェント実験 | シングルエージェント本番運用 |
| 「やってみた」記事 | 半自動でリリースし続けている事実 |

---

## 一般化: Aigis 以外でも使えるのか

「ドメイン × 段階 × 保留」の3つは、何の OSS でも書ける:

| OSS タイプ | ドメイン例 | 段階の中身 |
|---|---|---|
| **セキュリティスキャナ** | 攻撃カテゴリ 10 種 | 論文→regex→テスト→リリース |
| **Linter** | 言語 / フレームワーク別 | 違反ケース調査→ルール→テスト→リリース |
| **ドキュメント生成** | API / Tutorial / FAQ | 不足点調査→生成→確認→リリース |
| **CLI ツール** | サブコマンド別 | UX 不満調査→改善→テスト→リリース |

「ドメイン定義ファイルを書き換えるだけで、あなたの OSS にも同じループを導入できる」が Plugin の理想像。

## やらないこと (本編スコープ外)

- Multi-agent / Adversarial loop (Aigis にはあるが本編では触れない、別記事で展開可)
- Reinforcement learning / Fine-tuning (Forge Loop はプロンプト + 規約だけで成立)
- 大規模分散実行 / Kubernetes 統合 (cron + Claude Code で十分)
- セキュリティドメイン特化 (Aigis は事例の1つ)
